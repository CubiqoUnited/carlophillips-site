# Deployment Workflow

## Branch And PR Flow

1. Create a feature branch.
2. Make code changes.
3. Run local checks, at minimum `npm run build`.
4. Open a GitHub PR.
5. Let GitHub Actions run.
6. Let Vercel create a preview deployment.
7. Perform Desktop Codex browser QA against local and/or preview URLs.
8. Fix issues in the same PR.
9. Re-verify.
10. Merge only after user approval.
11. Deploy production from `main`.
12. Perform live storefront smoke test.
13. Confirm rollback option after deploy.

## Production Guardrails

- Do not merge/deploy production without approval.
- Do not directly change production env vars without approval.
- Production deploys should come from `main`.
- Main should require PR, CI/build, and Vercel preview before merge.

## Vercel Linking

Local Vercel operations require CLI authentication and a local `.vercel/project.json` generated by `vercel link`.

Do not commit `.vercel/`. After linking, verify the selected project/org before running preview, env, or deployment commands.

## Preview QA

For protected previews, use:

```bash
vercel inspect <deployment-url>
vercel curl / --deployment <deployment-url>
```

The Vercel CLI can generate a temporary deployment protection bypass for the linked project after login.

See `docs/vercel-env-setup.md` before adding or changing Vercel environment variables.

## Rollback

Rollback should use Vercel's previous successful production deployment unless a code revert is specifically safer. Document rollback action and timestamp in `docs/codex-shared-notes.md`.
